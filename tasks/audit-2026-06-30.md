## Behobene Bugs (mit Severity)

### CRITICAL — Money-Loss-Risiko

| ID | Bug | Fix-Datei |
|----|-----|-----------|
| D | `/api/finance/transfers` — 2 sequentielle Balance-RPCs (Network-Failure zwischendurch → Geld weg) | `backend/src/lib/helpers/finance.ts` neuer `transferBetweenAccounts` Helper, `routes/finance/index.ts` nutzt ihn |
| F | `createPeerTransfer` in `group-shared.ts` — gleiches Problem | `backend/src/lib/helpers/group-shared.ts` neue Atomicity + Ledger-Rollback bei Fehlern |
| H | RPC `increment_bank_balance` war NICHT in migrations enthalten — wäre stillschweigend gescheitert | `seeds/migrations/2026-06-30_audit_fixes.sql` CREATE OR REPLACE |
| G | `createPeerTransfer` (group-shared) erzeugte keine paired `private_expenses`/`income` Rows → Money landete auf Konto, aber kein Eintrag im Buchungs-Journal | Jetzt werden paired ledger rows mit `transfer_id` tag erzeugt |
| L | **`.catch()` auf QueryBuilder warf TypeError zur Laufzeit** — QueryBuilder hat nur `then`, kein `catch`. Alle Rollback-Pfade waren defekt. | Alle `.catch(() => undefined)` entfernt; der DB-Client schluckt Errors bereits intern via `{data, error}` return |
| M | Funding-Donate Rollback war race-unsicher (absolute amount-Reset → wipte parallel-Donors aus) | Neue `refund_from_funding` RPC mit `FOR UPDATE` + delta-decrement |
| N | Funding-Donate: letzte `increment_bank_balance` RPC ohne Error-Check → User behielt Pool-Eintrag aber Balance unverändert (invented money) | Balance-RPC checkt jetzt error; bei Fehler full rollback aller vorhergehenden inserts |

## Manuelle Schritte für den User (PFLICHT vor Code-Test)

### 1. Migration ausführen

Öffne den **Supabase SQL Editor** und führe diese Datei aus:
```
seeds/migrations/2026-06-30_audit_fixes.sql
```

Sie ist **idempotent** — kann mehrmals ausgeführt werden ohne Schaden.

Sie macht:
- `ALTER TABLE group_funding ALTER COLUMN group_activity_id DROP NOT NULL`
- `CREATE OR REPLACE FUNCTION increment_bank_balance(p_account_id INT, p_delta NUMERIC)`
- `CREATE OR REPLACE FUNCTION transfer_between_accounts(p_from_id INT, p_to_id INT, p_amount NUMERIC)`
- `CREATE OR REPLACE FUNCTION contribute_to_funding(p_funding_id INT, p_amount NUMERIC)`
- `CREATE OR REPLACE FUNCTION refund_from_funding(p_funding_id INT, p_amount NUMERIC)` (race-safe rollback)

### 2. Verifiziere im Supabase Dashboard

Im SQL-Editor:
```sql
SELECT proname FROM pg_proc WHERE proname IN (
  'increment_bank_balance', 'transfer_between_accounts',
  'contribute_to_funding', 'refund_from_funding',
  'release_period_reservations'
);
```

Sollte **5 Zeilen** zurückgeben.

### 3. Smoke-Test (Browser)

Voraussetzung: 2 Test-User registriert, beide mit mindestens 1 Bankkonto und gesetztem **Standardkonto** (Accounts → "Als Standard"). Beide in derselben Gruppe.

2. **Erwartung**: Toast "Nur 20,00 € wurden angenommen — Ziel erreicht." Sammelaktion-Status → "completed". Balance runter um 20€, nicht 100€.

#### Gruppenausgabe Prepaid (Bug A, B - jetzt funktioniert)

## Bekannte Issues (Out of Scope für diesen Audit)

- **Funding-Participants Race**: Wenn derselbe User exakt gleichzeitig zweimal spendet (extrem unwahrscheinlich in einem 1-User-Frontend), kann ein UNIQUE-Constraint-Violation auf `(group_funding_id, bank_account_id)` auftreten. Frontend zeigt dann unsere klare Error-Message. Kein Money-Loss.
- **Trip-Settlement-Recompute bei offenen Settlements**: Wenn Admin eine Ausgabe löscht während ein Settlement gerade beglichen wird, könnte das Settlement gelöscht und neu erstellt werden. Edge-Case, nicht User-reported.
- **Funding-Donate atomicity**: Drei separate DB-Operations (RPC + participant insert + expense insert) — Rollback ist best-effort via Compensating Updates. Keine echte 2PC-Transaktion. Bei Postgres-Crash kann inkonsistent bleiben — aber Money-Saldo ist sicher, weil Balance erst NACH dem expense-Insert decremented wird.
